Integration Test Framework · Idea Pitch

CTF — the Contract Test Framework

Stop wiring tests to machines. Declare what a test needs as a contract, describe when things come alive as a graph — and let the framework compose the environment deterministically for any target.

WHAT · composition plane WHEN · lifecycle plane contracts = the only coupling
Act I · The pain

Traditional test frameworks wire tests to machines, not to needs

What a test actually says today

# "an SSH test" — but look what it really hardcodes
def test_reboot():
    ssh = SSHClient("10.0.0.5", port=22)   # which box
    ssh.deploy_token("token-42")          # how it's auth'd
    ssh.connect()                          # the transport
    ssh.run("reboot")                       # the only real intent

One line of intent, four lines of environment. The test is the wiring.

So every test inherits the setup's assumptions

  • Change the transport (UART instead of SSH) → rewrite every test
  • New target with the same capability → fork the suite
  • Auth differs per lab → conditionals leak into tests
  • Bring-up order (power → net → creds) → hand-orchestrated in fixtures
  • “Does this box support X?” → scattered if guards

The intent — “reboot the target” — is buried under a specific machine on a specific day.

Act I · Why it doesn't scale

Ecosystems break on four unfounded assumptions

1 · “The test knows the target”

Handles, IPs, and transports are baked into test code, so tests aren't portable — they're re-implemented per bench.

2 · “Capabilities imply a backend”

exec silently means SSH. The moment a target execs over a serial console, the abstraction collapses.

3 · “Setup order is a script”

Deploy-token-before-connect lives in imperative fixtures. Reorder one dependency and it silently breaks.

4 · “Plugins may inspect each other”

Plugin A imports plugin B to check “is it there?”. Now they ship together forever — the ecosystem is one repo pretending to be many.

Each is a coupling the framework never forced you to break — so nobody did.

Act I · The root cause

None of these are bugs — they're the orientation

The framework points at machines

Every primitive it hands you — a client, a fixture, a connection — is a thing on a bench. So the natural way to write a test is to name that thing. The tool makes coupling the path of least resistance.

It has no word for “a need”

There's no first-class way to say “I need exec, I don't care how.” Intent gets encoded as a concrete transport, and every test silently over-specifies.

Change the mental model, and the symptoms dissolve

SymptomRoot orientation
tests aren't portabletests name machines
backend swap = rewritecapability ≡ transport
fragile setup orderorder is imperative
plugins ship togethercoupling by import

Fix the orientation once — point tests at needs, not machines — and all four fall out together.

Act II · The concept

The fix is one inversion:

A test declares the capability it needs — a string contract.
The framework resolves who provides it, in what order, for this target.
Tests never name a machine, a transport, or a boot sequence again.

Decouple by contract

A string is the entire public API and the only cross-plugin link.

Compose by graph

Dependencies resolve topologically; order is a consequence, not a script.

Choose by composition

Which backend? Decided when you assemble the run — never at call time.

Act II · The concept

An ecosystem that's collaborative, composable, reusable

Collaborative

Independent teams and vendors publish plugins from their own repos. They agree on names — contract strings — the way web services agree on URLs. No shared codebase, no import graph.

Composable

A run is assembled, not coded: drop in the plugins you want and the graph wires them into a valid environment — or tells you exactly what's missing.

Reusable

A plugin is written once against a contract and works on every target that satisfies it — mock, container, or silicon — with zero edits.

The unit of sharing isn't a library or a base class — it's a named capability. That single choice is what makes the other three possible.

Act II · The concept

Two planes that meet at one line

WHAT · composition · nouns

What is the target made of?

collect → resolve → realize → DUT

Primitive: a Provider / Descriptor. One live provider per string (SSOT). A pure graph, realized eagerly, ordered by @requires.

WHEN · lifecycle · verbs

What is done to it, and when?

pluggy hooks that only require — never provide.

Primitive: an action (provision, sanity-check, before/after test). Fanout, side-effecting, fired after the graph is fully realized.

They meet here

@hookimpl
def ctf_provision(ctx):        # a VERB — no provide
    ctx.require("ctf/cap/exec").execute("seed.sh")

A lifecycle action asks the composition plane for an already-built capability by contract string. That single require is the whole seam — and it's a warm cache hit.

Act II · The concept

Every contribution is a noun or a verb — never both

Producer — a noun

Answers “what is X?”. Resolves to a value someone can require. Exactly one per string (SSOT), realized lazily, ordered by its edges.

@provides("ctf/sec/token")      # yields a value
@requires("ctf/sec/token_request")
def deploy_token(req):
    deploy(req); yield Credential(req)

Action — a verb

Answers “do X”. Returns nothing; only side-effects. Many allowed (fanout), require-only, fired after the graph is ready.

@hookimpl
def ctf_provision(ctx):       # returns nothing
    ctx.require("ctf/cap/exec").push("seed.db")

The test for anything you register: does someone require your result? Yes → noun (SSOT binds you). No → verb (multiply freely). SSOT governs nouns; it was never a rule about verbs.

Act III · What guides every decision

Five pillars — each with a design approach and a choice it forces

1 · Decouple by contract

Sharing is a string, never an import. choice · identity, not shape

2 · Order is a derived tier

Phase is the computed topological layer. choice · derived tiers, no phase tags

3 · One truth per contract

Exactly one live provider, picked at compose time. choice · single source of truth

4 · Reuse by target-blindness

Plugins name facts, never a target. choice · descriptors + requires

5 · Availability is a graph fact

“Can this run?” is resolved, never guessed. choice · loose mode, no self-inspection

The next five slides take each pillar one at a time.

Act III · Pillar 1

A contract is a string — identity, not shape

@provides("ctf/cap/exec")          # I produce this
@requires("ctf/net/ssh_endpoint")   # given this
def ssh_exec(endpoint):
    return SshExec(endpoint)
  • Identity — “which thing do I mean?” The registry keys on it.
  • Zero coupling — no shared import; repos agree on a name, like an HTTP path.
  • Uniform — capabilities and facts (endpoints, tokens) are all just strings.

Why not a Protocol / interface?

  • Shapes collide — restart() and poweroff() are both ()→None
  • To “require an interface” every repo must import the class → the coupling returns
  • Facts have no methods — a tuple (host, port) can't be a Protocol
  • You'd have to instantiate to match a shape — but the graph is planned before anything runs

Protocols stay optional shape docs for mypy — never the resolution key.

The tier law

A contribution may require only lower tiers. So two things in the same tier can't see each other — peer coupling isn't forbidden, it's impossible. “Depend only on the target” is just “the target is tier 0.”

Tiers are derived, never declared

No phase= on any provider. Each @requires edge places a node in its tier; the ladder is the toposort, rendered. If order matters, an edge exists — and the dependent thing is a higher tier by construction.

All @provides("ctf/cap/exec"). The composition registers exactly one. Two live at once → DuplicateProviderError, by design.

“Which backend?” is never a runtime guess

  • By knob — --ctf-target=docker registers the docker exec
  • By graph — the backend whose transport fact exists resolves; the rest are simply unavailable
  • By name — if a console is not substitutable for SSH, it's a different contract (ctf/console/exec)

The test asks require("ctf/cap/exec") and never learns which one it got.

Act III · Pillar 4

Reuse comes from target-blindness

# a plugin NEVER names a target —
# it names the FACTS it consumes
@provides("ctf/cap/exec")
@requires("ctf/net/ssh_endpoint")  # a fact
def ssh_exec(endpoint):
    return SshExec(endpoint)

Design approach: targets publish facts (descriptors); capabilities consume them. The two never reference each other — only a shared string.

Why it's reusable everywhere

  • The same ssh_exec serves a sealed box, an open box, a container — any target that publishes an ssh_endpoint
  • Co-located by file ≠ coupled by contract — a target and its plugin can ship from different repos
  • Add a new target: publish the fact, reuse every capability unchanged

Choice it forces: no import target, no target-specific handles — only descriptors and @requires.

Act III · Pillar 5

Availability is a graph fact, never a self-check

The old way — plugins inspect each other

if has_plugin("token"):        # import + guard
    endpoint = authed()
else:
    endpoint = anon()

Guards scatter, plugins import each other, and “is X here?” is answered at runtime, differently in every plugin.

The CTF way — the graph answers it once

  • A capability resolves iff its required facts exist in the registry
  • Missing dependency in LOOSE mode → recorded unavailable; the test simply skips
  • No plugin ever calls registry.has() to gate itself

Design approach: presence is a property of the resolved graph, computed centrally. Choice it forces: loose-mode resolution + honest skips instead of scattered if guards.

Act V · Architecture — zoom 2 · the WHEN plane

Lifecycle — pluggy collects, the engine resolves

Split by kind of citizen — never by phase. Phase is derived by the engine, not chosen by a hook.

Hooks require, never provide

@hookimpl
def ctf_provision(ctx):        # a verb: act, return nothing
    link = ctx.require("ctf/net/link")   # warm — already built
    push_config(link, "app.yaml")
  • collect — pluggy gathers a flat bag of declarations; orders nothing
  • resolve — the engine builds the graph, enforces SSOT, toposorts into tiers
  • realize — eager, fail-fast; by the time any hook fires the cache is warm
  • Because a hook can't provide, load-bearing order can never hide in a hook — it's always a graph edge

Pluggy is a collection bus, not a scheduler. Discovery is pluggy's; ordering, resolution & execution are the graph's — so the runner is swappable.

Act VI · Put it to work

One capability. Two very different targets.
Zero test changes.

Scenario: run a command over SSH. Target A is a sealed box whose access needs a deployed key. Target B is an open box that needs none. Watch the environment reshape itself while the test stays still.

Act VI · Step 1 — targets declare facts

Targets publish facts + a linked endpoint, nothing else

Target A — sealed (needs a key)

TOKEN = "ctf/sec/token"
@provides("ctf/net/ssh_endpoint")
@requires(TOKEN)                # the LINK
def endpoint(credential):
    return ("10.0.0.5", 22, credential)

# fact: which key this box needs
add(Descriptor("ctf/sec/token_request", "token-42"))

Access is gated: the endpoint can't integrate until the key deploys.

Target B — open (needs nothing)

@provides("ctf/net/ssh_endpoint")
# no @requires -> resolves standalone
def endpoint():
    return ("10.0.0.9", 22, None)

“Don't declare the requirement, and it resolves without the dependency.” The credential slot is simply None.

Neither target imports the SSH plugin, the token plugin, or each other. The only link is the contract string.

Act VI · Step 2 — reusable plugins

One SSH plugin. One key deployer. Both target-blind.

SSH exec — needs only an endpoint

@provides("ctf/cap/exec")
@requires("ctf/net/ssh_endpoint")
def ssh_exec(endpoint):
    host, port, cred = endpoint
    return SshExec(host, port, cred)

Knows nothing about tokens. Whether a key was required is the target's business — carried inside the endpoint value.

Key deployer — keyed on a fact

@provides("ctf/sec/token")
@requires("ctf/sec/token_request")
def deploy_token(token_id):
    deploy(token_id)
    yield Credential(token_id)
    revoke(token_id)          # reverse-order teardown

Resolves only where a token_request fact exists. On Target B it's simply unavailable — no error, nothing needs it.

Act VI · Step 3 — the test consumes

The test asks for a capability and runs. That's all.

    assert code == 0

No IP. No transport. No auth. No boot order. The test states its need and consumes the result.

Act VI · Provisioning — two deploys that look identical

Same verb “deploy”, two different citizens

Deploy a token → a noun

@provides("ctf/sec/token")        # yields Credential
@requires("ctf/sec/token_request")
def deploy_token(req):
    deploy(req); yield Credential(req); revoke(req)

# ssh_endpoint @requires("ctf/sec/token")

ssh_endpoint consumes the credential → it's a producer, in the graph. The edge guarantees key-before-endpoint; “ssh fails if no token” is made unrepresentable.

Deploy seed data → a verb

@hookimpl
def ctf_provision(ctx):          # returns nothing
    sh = ctx.require("ctf/cap/exec")  # warm
    sh.push("seed.db", "/var/app/")
    sh.execute("app --load app.yaml")

Nothing constructs a capability from seed data → it's a post-READY action. Fanout, order-free, no SSOT. It just has to exist before tests run.

The tell is mechanical: does a @provides factory read your result? Yes → graph noun (ordered by edges). No → ctf_provision verb (post-ready fanout). You decide “consumed?”, and the citizen picks itself.

Act VI · Pattern 1 — unit

Unit tests — the framework gets out of the way

# test_parser.py — no target, no contracts, no composition
from mylib.parser import parse

def test_parse_roundtrip():
    assert parse("a=1;b=2") == {"a": 1, "b": 2}

Plain pytest. No ctf/target anchor, no providers registered.

Zero tax when you don't need it

  • With no anchor and no providers, the composition plane is inert — nothing to resolve.
  • You import the code under test directly and assert. Same as any pytest suite.
  • The moment you need a real target, you add a contract — you never rewrite the test's intent.

Escalation is additive: unit → component is “introduce an anchor”, not “migrate frameworks”.

Act VI · Pattern 3 — integration

Integration — deploy a full image, talk to it over interfaces

# the image is a FACT (a descriptor) — tier 0
add(Descriptor("ctf/image", "build/system.img"))

# flashing brings the target into existence — tier derived from its edges
@provides("ctf/target")
@requires("ctf/image")              # image → target
def flashed_target(image):
    flash(image); boot()
    yield Board()
    power_off()               # reverse-order teardown

Deploy happens because ctf/target requires the image — not because a fixture called flash() first.

Then interact via interfaces — oblivious to flashing

def test_service_up(shell, files):
    files.push("cfg.yaml", "/etc/app/")
    code, _ = shell.execute("systemctl is-active app")
    assert code == 0

The test consumes exec + file_transfer capabilities. It never sees the flash, boot, or power-off — the ladder handled all of it.

Act VI · Pattern 4 — provisioning & liveness

Bench provisioning + a heartbeat that keeps the target reachable

TIER 0

TIER 1

@provides("ctf/bench/power")
def energize():
    tb.power_on(); yield; tb.power_off()

@provides("ctf/target")
@requires("ctf/bench/power")      # boots only once powered → higher tier
def board(power): ...
@provides("ctf/net/link")
@requires("ctf/target")            # target → link, tier follows
def heartbeat(target):
    hb = start_keepalive(target)   # background thread
    yield hb
    hb.stop()                       # stops on teardown

# exec @requires("ctf/net/link") -> usable only while alive

Bench powers the board (tier 1), the heartbeat holds it reachable (tier 2), capabilities light up (tier 3) — order and liveness both fall out of the graph, and unwind in reverse.

Act VI · Pattern 3½ — messy hardware

Recovery — invalidate a noun, rebuild lazily

# heartbeat dropped mid-test; the box is still up but unreachable
dut.invalidate("ctf/net/link")   # link + everything above it,
                              # torn down in reverse order

# rebuild is lazy — just ask again
shell = dut.require("ctf/cap/exec")  # fresh, over a live link

The DUT handle a test holds outlives any one composition. You recover by re-requiring, never by mutating a stale object in place.

What invalidate does

  • Drops the node and its transitive dependents — the reverse-dependency closure
  • Tears them down in reverse instantiation order — no stale handle survives over a re-flashed box
  • Rebuild is on-demand: the next require re-realizes the subtree in tier order

Why a fixture can't do this

pytest's fixture cache is compute-once and opaque. There is no “drop this node and rebuild the subtree” — an autouse fixture fires once, and you can't re-fire it. This single requirement is why the engine owns its own cache.

Honest residue

  • Re-flash invalidates a token deployed on the old firmware → reverse teardown runs revoke-then-redeploy for you
  • Verbs re-fire by re-requiring their nouns; provisioning-of-state is a noun, so it comes back on rebuild
  • Recovery is explicit — a test calls invalidate. No hidden watchdog; failure detection stays domain code
  • Finalizers must be defensive (idempotent) — an upstream node may already be gone
Act VI · Pattern 4½ — ordering without peer coupling

Dev-flash then eng-flash — order via a state contract, not a plugin

The wrong edge — peer coupling

@provides("ctf/fw/signed")
@requires("ctf/flash/dev")   # ✗ names a PEER
def eng_flash(dev): ...

eng now can't exist without the dev plugin loaded. Two contributors bound to each other — exactly the coupling the tier law forbids.

The right edge — a state

@provides("ctf/fw/signed")
@requires("ctf/fw/base")     # ✓ a STATE it needs true
def eng_flash(base): ...

eng requires base firmware present — owned by the target surface, not by whoever happens to establish it.

Who satisfies ctf/fw/base? — establish vs assert

# CI: actually do the work (hermetic)
@provides("ctf/fw/base")
def dev_flash():
    flash_dev(); yield BaseFw()

# light run: assert it's already there
add(Descriptor("ctf/fw/base", present=True))
  • Establish — a producer does the flashing (reproducible pipeline)
  • Assert — a descriptor swears the board already has it (fast bench run, no dev import)
  • SSOT: exactly one provider of ctf/fw/base per run
  • Neither present → honest compose error / skip, never a silent wrong-firmware pass

eng-flash is written once and doesn't care which way base got there — it only requires the state.

Act VI · Pattern 5 — one plugin, three owners

UDS over DoIP and/or CAN — the UDS plugin owns none of the wires

Three independent owners: the target exposes facts, a transport plugin turns a fact into a byte-mover, the UDS plugin speaks UDS over whatever transport it's handed. One UdsClient, re-exported per transport it can bind to.

Shape A · “either wire is fine” — substitutable

# ONE transport noun, two backend variants
@provides("ctf/diag/transport", variant="can")
@requires("ctf/bus/can")              # target fact
def can_tp(bus): ...

@provides("ctf/diag/uds")
@requires("ctf/diag/transport")       # a NOUN, not a wire
def uds(tp): return UdsClient(tp)

Composition selects one transport variant → one live uds. Two live at once → DuplicateProviderError.

Shape B · “test both wires” — simultaneous

# the UDS plugin re-exports ONE impl per transport
@provides("ctf/diag/uds/can")
@requires("ctf/link/can")             # another plugin's contract
def uds_can(link):  return UdsClient(link)

@provides("ctf/diag/uds/doip")
@requires("ctf/link/doip")
def uds_doip(link): return UdsClient(link)
@parametrize("uds", ["ctf/diag/uds/can",
                        "ctf/diag/uds/doip"])
def test_read_vin(dut, uds):
    assert dut.require(uds).read(0xF190)

Whichever transports resolve become the available UDS variants; the test chooses. Both reuse the same UdsClient.

The disambiguator — ask the test's intent, not the device: one result over a chosen path → one ctf/diag/uds noun, transport selected (Shape A). two results to check independently → two distinct uds/can, uds/doip nouns, both live (Shape B). A device that physically supports both never picks — intent does.

Act VI · How to compose any of them

The composition recipe — five moves

1 · Choose the anchor

No ctf/target → unit. One anchor → a device under test. The anchor decides mode.

2 · Publish facts

Targets declare what they have: endpoints, buses, images, key-requests — plain descriptors, string-keyed.

3 · Register backends

Capabilities @provide a contract and @require the facts they consume. Reusable, target-blind.

4 · Let tiers derive themselves

You never tag a phase. Each @requires edge places a node in its tier; the ladder is the toposort rendered. Order that matters is an edge; order that doesn't is fanout.

5 · Tests require by string

A test names the capability it needs and nothing else. The graph resolves who, what, and when.

That's the whole method

Every pattern here — unit to UDS-over-CAN — is these five moves with different facts. You compose; you never orchestrate.

Act VII · The honest question

Why not just pytest fixtures?

Fixtures already give you ~80%

  • Lazy, cached, dependency-ordered, reverse-teardown
  • A noun ≈ a session fixture; a verb ≈ a hook
  • Honest: this is why the engine rides pytest for collection, test items, parametrization & reporting

If that were the whole job, the engine would be redundant. It isn't — for five reasons pytest's static, per-name cache can't express.

The 20% the engine owns

  • SSOT error, not silent fixture override — two providers of one contract is a hard stop
  • Eager fail-fast at compose time, not lazy-per-request deep in a test
  • String namespace + target-blind selection, not name + conftest-proximity
  • The tier law is structural — derived from the graph, not a convention you hope holds
  • Recovery — invalidate one node, rebuild its subtree, re-fire its verbs

The decider: pytest's fixture cache is compute-once and opaque. The moment you must re-flash a box mid-run and rebuild only what sat on top of it, you need a cache you can invalidate — so the engine owns one.

Act VII · The payoff

What the inversion buys you

Portable tests

A test targets a capability, so it runs on any box that provides it — mock, docker, or real silicon.

Swappable backends

SSH today, UART tomorrow: add a provider, pick it in composition. Tests untouched.

Order without scripts

Deploy → connect → provision falls out of the dependency graph, and unwinds itself.

True multi-repo

Plugins share only strings. No plugin imports another; each can ship from its own repo.

Deterministic

One provider per contract; a fixed plan for a fixed registry. No hidden tie-breaks.

Honest availability

Missing capability = a graph fact → the test skips. No scattered if guards.

Tests declare needs.
The graph composes the environment.
Contracts are the only thing anyone shares.

Everything a traditional framework hardcodes — the box, the transport, the auth, the boot order — becomes a resolvable contract. Write the test once; run it everywhere.
